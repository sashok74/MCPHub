# /pm-save-task - Сохранить решённую задачу в project-memory

Сохраняет описание проблемы, причину и решение в базу знаний проекта для будущего поиска.

## Использование

```
/pm-save-task                    # Автоматически собрать из контекста беседы
/pm-save-task [описание]         # Сохранить с явным описанием
```

## Алгоритм выполнения

### Фаза 1: Сбор информации из контекста

Проанализируй последние сообщения беседы и извлеки:

1. **Проблема** — что было сломано, симптомы, как проявлялось
2. **Причина** (root cause) — ПОЧЕМУ это происходило
3. **Решение** — что конкретно сделали (с кодом если есть)
4. **Паттерн/Gotcha** — общий урок, применимый к другим местам
5. **Файлы** — какие файлы были изменены
6. **Issue номер** — если есть (#1592, qFix и т.д.)

### Фаза 1.5: Извлечение цепочки рассуждений

Проанализируй ВСЮ историю беседы и восстанови диагностический маршрут:

1. **Начальная гипотеза** — что предположили первым
2. **Шаги проверки** — что искали, какие инструменты использовали, что нашли
3. **Тупики** — что НЕ сработало и почему (ценнее всего!)
4. **Прорыв** — момент, когда стала ясна настоящая причина
5. **Верификация** — как убедились что решение правильное

Формат каждого шага:
```
N. [Действие] -> [Результат/Находка]
```

Ограничения:
- Максимум 5-8 шагов (конденсировать, не дамп)
- Каждый шаг — 1-2 строки
- Фокус на РЕШЕНИЯХ и ПОВОРОТНЫХ ТОЧКАХ, не на рутине
- Пропускать тривиальные шаги (чтение файла, компиляция)

### Фаза 2: Формирование описания

Описание задачи должно быть структурированным:

```markdown
## Проблема
[Что сломалось, симптомы — 1-3 предложения]

## Причина
[Root cause — почему это происходило]

## Решение
[Что сделали — с ключевыми фрагментами кода]

## Паттерн
[Общий урок / gotcha — чтобы не повторить ошибку]

## Файлы
- [список изменённых файлов]

## Цепочка поиска решения
1. [Гипотеза/Действие] -> [Результат]
2. ...
```

### Фаза 3: Сохранение

```
save_task(
  title="Краткое описание задачи",
  description="## Проблема\n...\n## Решение\n...",
  issue_number="#номер",      # опционально
  author="Claude + User"      # или другой автор
)
```

### Фаза 4: Верификация

```
search_tasks(query="ключевое слово из задачи")
```

Убедиться что задача находится по разным ключевым словам (симптом, причина, решение).

## Принципы хорошего описания

### Title (заголовок)
- Краткий, но информативный
- Включает модуль/сущность + суть проблемы
- Примеры:
  - "Исправление редактирования версии формы OrderForm"
  - "Ошибка фильтрации combo в форме выбора"
  - "Переполнение при выборке данных с большим количеством записей"

### Description (описание)
- **Проблема**: опиши СИМПТОМ — что видел пользователь
- **Причина**: опиши ROOT CAUSE — почему это происходило технически
- **Решение**: опиши ЧТО СДЕЛАЛИ — с ключевым кодом (не весь diff, а суть)
- **Паттерн**: опиши ОБЩИЙ УРОК — применимый к другим ситуациям

### Поисковые ключи
Описание должно содержать достаточно ключевых слов для поиска:
- Имена классов, методов, таблиц
- Симптомы ("портится", "перезаписывает", "не сохраняется")
- Технические термины ("paramMap", "SaveDataFromControls")

## Поиск по задачам

```
search_tasks(query="ключевые слова")           # FTS поиск по title+description
search_tasks(author="Claude + User")            # по автору
search_tasks(date_from="2026-02-01")            # по дате
search_tasks(query="Orders", limit=5)           # с лимитом
```

## Когда использовать

- После решения бага или issue
- После нетривиального исправления с gotcha
- После рефакторинга с важными уроками
- Когда обнаружен паттерн, который может повториться
- НЕ нужно для тривиальных правок (опечатки, форматирование)

## Пример

```
/pm-save-task
-> Анализ контекста... Найдена решённая задача:
-> save_task(
    title="Исправление редактирования версии формы OrderForm",
    description="## Проблема
При сохранении формы портились ActionID и StateID — перезаписывались значениями из paramMap.

## Причина
SaveDataFromControls() записывал значения combo-контролов поверх полей, которые при определённых условиях должны быть read-only. paramMap не исключал ActionID/StateID для этого случая.

## Решение
Добавлена проверка IsReadOnly() перед записью ActionID/StateID в SaveDataFromControls(). Если условие выполнено — поля пропускаются.

## Паттерн
В CBaseCrudDialog.SaveDataFromControls() paramMap применяется ко ВСЕМ полям без условий. Если поле должно быть read-only при определённых условиях — его нужно исключать из маппинга динамически, а не просто делать контрол disabled.

## Файлы
- OrderFormDlg.cpp

## Цепочка поиска решения
1. Гипотеза: combo контролы записывают значения при сохранении -> Подтверждено: SaveDataFromControls() пишет ВСЕ поля из paramMap
2. Поиск: где формируется paramMap для ActionID -> InitializeControlMapping() — безусловный маппинг
3. Тупик: попытка убрать контрол из маппинга в OnInitDialog -> не сработало, маппинг статический
4. Прорыв: нужно условие ВНУТРИ SaveDataFromControls — проверка IsReadOnly() перед записью
5. Верификация: компиляция + тест: форма сохраняется без порчи ActionID/StateID",
    issue_number="#qFix",
    author="Claude + User"
  )
-> Задача сохранена (id=1)
-> Проверка: search_tasks("OrderForm версия") -> найдена
```
