# /pm-remember - Кураторское сохранение знаний в project-memory

Сохраняет в project-memory только проверенные и реально переиспользуемые знания из текущего исследования. Команда не должна слепо майнить последние сообщения беседы и не должна сохранять выводы без evidence.

## Использование

```
/pm-remember                    # Сохранить curated findings из текущего исследования
/pm-remember [что сохранить]    # Сохранить конкретную evidence-backed информацию
```

## Что делает команда

1. **Собирает evidence-backed находки из текущей задачи**
   - код, который был прочитан через `Read`/`Grep`
   - результаты `dbmcp`
   - проверенные SQL-запросы
   - реальные UI-наблюдения
   - документацию, которую действительно открывали
2. **Отфильтровывает мусор**
   - не сохраняет выводы только из ответа ассистента
   - не сохраняет результаты `prepare_context`/`ask_business` без верификации
   - не сохраняет временные догадки, черновики и одноразовые debug-наблюдения
3. **Проверяет дубликаты**
   - предпочитает `update_fact`/`delete_fact`, а не новые дубли
4. **Сохраняет только то, что будет полезно в будущих сессиях**
   - schema/business_rule/gotcha/form_mapping/glossary/query/relationship/process
5. **Выдаёт отчёт**
   - что именно сохранено
   - что сознательно не было сохранено и почему

## Когда использовать

Используй `/pm-remember` только в этих случаях:
- пользователь явно попросил сохранить знания в PM
- завершилось отдельное исследование, и есть 3+ проверенных, переиспользуемых находки
- найдено важное соответствие form/template/class/table, которое точно пригодится снова
- найдено устойчивое бизнес-правило, связь, проверенный SQL или термин с evidence

Не использовать автоматически:
- после каждого `dbmcp` вызова
- после каждого чтения `doc/*.md`
- после любого ответа "что такое X?"
- просто потому что "надо что-то сохранить"

## Workflow для Claude

```
1. Claude завершил исследование и собрал доказательства
2. Claude отбирает только стабильные знания, полезные в будущих сессиях
3. Claude проверяет, нет ли уже такого факта/маппинга/связи
4. Claude сохраняет curated-набор
5. Claude показывает отчёт: что сохранено и что отброшено
```

## Примеры использования

### Пример 1: После реального исследования
```
User: расскажи про таблицу Orders
Claude: [выполняет dbmcp:get_table_schema, get_table_relations, читает код]
Claude: [отвечает пользователю]
Claude: /pm-remember
-> Сохранено:
  - 3 facts (schema, business_rule, relationship)
  - 1 glossary (Заказ)
  - 1 form_mapping (Order -> COrderDlg)
```

### Пример 2: Явное указание пользователя
```
User: /pm-remember Orders это документ заказов клиентов
-> Сохранено только если есть evidence в текущем исследовании.
-> Если evidence нет: "Нет новых доказанных знаний для сохранения"
```

### Пример 3: Когда сохранять НЕ надо
```
Claude: [получил prepare_context("Orders")]
Claude: [ещё не читал код и не делал dbmcp]
Claude: /pm-remember
-> Ничего не сохранять: prepare_context сам по себе не является evidence
```

## Что сохранять и куда

| Тип информации | Куда сохранять | Тип/Параметры |
|----------------|----------------|---------------|
| Структура таблицы БД (поля, типы, PK/FK) | facts | schema |
| Бизнес-правила, статусы, enum'ы | facts | business_rule |
| Связи между таблицами (FK, текст) | facts | relationship |
| **Направленные связи (A создаёт B)** | **relationships** | **rel_type: creates/contains/references/depends_on/produces/consumes** |
| Архитектурные паттерны (Entity, Template) | patterns | - |
| Формы и классы (dialog, viewform) | form_mappings | - |
| Термины (заказ, перемещение) | glossary | - |
| SQL запросы (проверенные) | queries | - |
| Пошаговые workflow | processes | - |
| Важные особенности, gotchas | facts | gotcha |
| Статистика (количество записей) | facts | statistics |
| Примеры кода, usage patterns | facts | code_pattern |
| **Группировка сущностей по модулям** | **modules** | **save_module(module_name, entities[])** |
| **Уровень изученности сущности** | **entity_status** | **coverage: unknown/schema_only/partial/full/needs_review** |

## Правила сохранения

### DO (что нужно делать):
- Сохранять только после верифицированного исследования
- Включать evidence (путь к файлу, номер строки)
- Помечать confidence: verified / unverified / uncertain
- Связывать сущности через `save_relationship(entity_from, entity_to, rel_type)` — структурированные связи
- Использовать Cyrillic свободно (поддерживается морфология)
- Сохранять синонимы для терминов
- Указывать menu_path для форм
- Группировать сущности по модулям: `save_module("Orders", ["Order", "OrderItem"])`
- Обновлять статус после исследования: `set_entity_status(entity, coverage="full")`
- Использовать `update_fact(id, ...)` для исправления неточных фактов вместо создания дубликатов
- Использовать `delete_fact(id)` для удаления устаревших фактов
- Использовать `bulk_save(facts=[...], glossary=[...], relationships=[...])` для пакетного сохранения
- Отбрасывать непереиспользуемые детали, одноразовые debug-наблюдения и догадки

### DON'T (чего избегать):
- Не майнить последние 5-10 сообщений беседы как самостоятельный источник истины
- Не дублировать существующие факты (проверить через search_facts; исправлять через update_fact)
- Не сохранять очевидные вещи без контекста
- Не забывать про glossary при новых терминах
- Не сохранять непроверенный код или SQL
- Не сохранять данные только из `prepare_context`, `ask_business` или ответа ассистента
- Не сохранять знания без evidence
- Не забывать про relationships при обнаружении связей между сущностями
- Не забывать обновлять entity_status после исследования

## Проверка сохранения

После /pm-remember всегда показывать отчет:
```
Сохранено в project-memory:
  * 4 facts (Orders: schema, business_rule, relationship, code_pattern)
  * 1 glossary (Заказ)
  * 1 form_mapping (Orders -> COrderDlg)
  * 2 relationships (Orders ->[contains] OrderItems, Orders ->[references] Customers)
  * 1 module (Orders += OrderHistory)
  * 1 entity_status (Orders: full)

Можно найти:
  - search_facts(query="Orders")
  - get_glossary(term="Заказ")
  - find_form(table="Orders")
  - get_relationships(entity="Orders")
  - get_entity_status(entity="Orders")
```

## Интеграция с другими командами

- После `/explore` -> использовать `/pm-remember` только если появились evidence-backed reusable findings
- После чтения `doc/*.md` -> не сохранять автоматически; сначала проверить, есть ли новая устойчивая информация
- После серии dbmcp запросов -> не сохранять автоматически; сначала сверить с кодом/контекстом задачи
- Перед завершением сессии -> сохранить только curated набор, а не весь контекст подряд

Все данные персистентны между сессиями Claude Code.
