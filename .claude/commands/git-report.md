# Git Report Skill

Этот skill генерирует развёрнутые отчёты по Git-коммитам с **глубоким анализом** что было сделано и зачем.

## Параметры

- **branch** (по умолчанию: HEAD) - ветка для анализа
- **days** (по умолчанию: 7) - период в днях
- **--author=Name** (опционально) — фильтр по автору. Можно указать несколько раз. Если не указан — все авторы.

## Примеры вызова

```
/git-report                              # отчёт за 7 дней по текущей ветке, все авторы
/git-report main 14                      # отчёт за 14 дней по main
/git-report 30 --author="John Doe"       # отчёт за 30 дней по конкретному автору
```

## Инструкции для Claude

### Шаг 1: Разбор аргументов

Разбери аргументы из `$ARGUMENTS`:
- Если указан один аргумент с цифрами - это `days`
- Если указан один аргумент без цифр - это `branch`
- Если указаны два аргумента - первый `branch`, второй `days`
- `--author=Name` — извлечь имена авторов (может быть несколько)
- По умолчанию: `branch=HEAD`, `days=7`, авторы — все

### Шаг 2: Сбор коммитов и статистики

Выполни параллельно:

```bash
# Список коммитов (если авторы указаны — добавить --author для каждого)
git log <branch> [--author="Name1" --author="Name2"] --since="<days> days ago" --format="%H|%s|%ad|%an" --date=short

# Общая статистика за период
git diff --shortstat $(git log <branch> [--author=...] --since="<days> days ago" --format="%H" | tail -1)^..HEAD
```

### Шаг 3: Классификация коммитов

Раздели коммиты на две группы:

**Информативные** (понятно что сделано):
- Есть номер Issue (`#1234`, `Issue #1234`)
- Сообщение описывает суть изменения

**Неинформативные** (требуют глубокого анализа):
- Короткие сообщения: `fix`, `bugfix`, `update`, `refactor`
- Технические: `conflict`, `merge`, `wip`
- Непонятные: однословные или без контекста

### Шаг 4: Глубокий анализ неинформативных коммитов

**КРИТИЧЕСКИ ВАЖНО:** Для каждого неинформативного коммита выполни:

#### 4.1. Анализ diff
```bash
git show <hash> --stat --format=""
git show <hash> -p --format="" | head -80
```

#### 4.2. Поиск контекста в project-memory (если подключён)
```
mcp__project-memory__search_facts(query="<ключевые слова из изменённых файлов>")
mcp__project-memory__ask_business(question="<что делает этот компонент?>")
```

#### 4.3. Реконструкция смысла изменения

По diff определи:
- **Что было** → **Что стало** (конкретные изменения кода)
- **Почему** это могло понадобиться (баг? улучшение? новая функция?)
- **Какую проблему** это решает

**Примеры реконструкции:**

| Commit message | Diff показывает | Реальный смысл |
|----------------|-----------------|----------------|
| `ret err` | `!x.size() > 1` → `!(x.size() > 1)` | Исправлен приоритет операторов (баг) |
| `column width` | Добавлен `ForceResizeColumns()` | Грид не обновлял колонки при resize окна |
| `initial date` | `2000-01-01` → `текущая - 1 месяц` | Отчёт загружал данные за 26 лет |

### Шаг 5: Группировка результатов

Сгруппируй по:
1. **Issues** - коммиты с явным номером Issue
2. **Логическим задачам** - связанные по компоненту/файлам
3. **Багфиксы** - исправления ошибок без Issue
4. **Мелкие улучшения** - остальное

### Шаг 6: Формирование отчёта

```markdown
## Отчёт по коммитам: [дата_начала] — [дата_конца]

**Ветка:** <branch> | **Авторы:** <список или "все">
**Всего:** N коммитов | **+X** / -Y строк | **M файлов**

---

### Issue #XXXX: [Название]

**Проблема:** [Что было не так - понятным языком]

**Решение:**
- [Конкретное изменение 1]
- [Конкретное изменение 2]

**Результат:** [Измеримый эффект, если есть]

---

### [Название багфикса без Issue]

**Проблема:** [Описание бага]

**Причина:** [Техническая причина - кратко]

**Исправление:** [Что изменено]

---

### Сводка по категориям

| Категория | Количество | Влияние |
|-----------|------------|---------|
| Производительность | N | Критическое/Высокое/Среднее |
| Безопасность | N | ... |
| UI/UX | N | ... |
| Багфиксы | N | ... |
```

### Правила анализа

1. **Описание должно быть понятно менеджеру/заказчику**, а не только разработчику
2. **Всегда указывай "Проблема → Решение → Результат"**
3. **Группируй связанные коммиты** в одну запись
4. **Определяй компоненты** по путям файлов и именам классов

### Оценка влияния

| Влияние | Критерии |
|---------|----------|
| **Критическое** | Производительность 10x+, безопасность, потеря данных |
| **Высокое** | Новая функциональность, права доступа, интеграции |
| **Среднее** | UI/UX улучшения, удобство работы |
| **Низкое** | Мелкие багфиксы, косметика, рефакторинг |

### Обработка ошибок

- Если нет коммитов за период - сообщи об этом
- Если ветка не найдена - сообщи об ошибке
- Если diff слишком большой - анализируй первые 80 строк + `--stat`
